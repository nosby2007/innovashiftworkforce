# SSR / Prerendering

## What this is

The four public marketing routes — `/`, `/features`, `/pricing`, `/contact`
— are prerendered at build time into static HTML using Angular's classic
Universal-style server builders (`@angular-devkit/build-angular:server` /
`:prerender`). This is build-time static generation (SSG), not a live SSR
server: there's no Node process rendering pages on request in production.
Firebase Hosting keeps serving plain static files, same as before.

Everything else in the app — `/app/*`, `/admin/*`, `/login`, `/register`,
`/print/*`, `/platform/*` — is unaffected and stays client-side rendered
only, exactly as before this change.

## Why

The public site previously shipped an empty CSR shell: view-source showed
only a loading spinner, with the real hero copy, features list, and pricing
plans injected by Angular after the JS bundle loaded and ran. Search engine
crawlers that don't execute JavaScript (or execute it unreliably) saw
nothing useful. Prerendering fixes this — `curl` or view-source on any of
the four routes now returns fully-rendered HTML with the real page content,
plus the per-page `<title>`/meta description already set by `SeoService`.

## What's out of scope

- **No per-locale URLs.** Prerendering only renders the default (`en`)
  locale. There's no `/fr/features` route — see `docs/I18N.md` for the
  French-content limitation this leaves in place.
- **No SSR of the authenticated app.** `/app/*` and `/admin/*` remain pure
  CSR. Server-rendering them would require every service and component
  reachable from those routes to be SSR-safe (see "Gotchas" below), which
  is a much larger undertaking than four static marketing pages warranted.
- **No live SSR server in production.** `src/server.ts` (an Express server
  that can render any route on request) was generated by the `@angular/ssr`
  schematic and is kept in the repo for local dev (`npm run dev:ssr`,
  `npm run serve:ssr`) but nothing deploys or runs it — Firebase Hosting
  only serves the static files the `prerender` build produces.

## How it works

`npm run prerender` (`ng run innovacare-shift-frontend:prerender`) runs the
browser build, then boots the app in Node once per configured route,
capturing the rendered HTML. The `prerender` target in `angular.json` sets
`discoverRoutes: false` with an explicit `routes` array of exactly the four
public pages — Angular's prerender builder defaults to auto-discovering and
prerendering *every* statically-resolvable route in the whole router config
(all 48 routes across `/app` and `/admin`), so this override is required to
keep prerendering scoped to the public site.

Output lands in `dist/innovacare-shift-frontend/browser/`, alongside the
normal browser build: `index.html`, `features/index.html`,
`pricing/index.html`, and `contact/index.html` are the prerendered pages;
everything else (JS chunks, assets, `ngsw.json`, etc.) is the same output a
plain `ng build` produces. `firebase.json`'s `hosting.public` and
`capacitor.config.ts`'s `webDir` both point at this `browser/` subdirectory
(the classic server builder wants a separate `browser`/`server` output
split, which is why the path grew a segment).

CI (`ci.yml`) and the deploy workflow (`deploy.yml`) both run
`npm run prerender` instead of `npm run build`, so every merge to `main`
validates the SSR-safety of the whole app (see below) and every deploy
ships the prerendered marketing pages.

## Gotchas: the whole app gets bundled, not just the prerendered routes

The classic Angular Universal server builder does **not** tree-shake the
server bundle down to only the routes being prerendered. It bundles the
entire reachable application — every lazy route, every injectable — into
one `server/main.js`, because in principle the server needs to be able to
render any URL. In practice this meant SSR-incompatible code anywhere in
the app (not just the four marketing pages) could crash the prerender
build, even though those code paths are never reached by request during a
real deploy.

Two classes of problem surfaced and were fixed as part of landing this:

1. **Module-evaluation-time browser globals.** `import * as L from
   'leaflet'` (in `geofence-map.component.ts` and
   `admin-org-settings.page.ts`) pulled in Leaflet's own UMD registration
   code, which unconditionally touches `window` at module-load time — not
   inside any function, so guarding call sites doesn't help. Fixed by
   switching to a type-only import (`import type * as Leaflet from
   'leaflet'`) plus a runtime `await import('leaflet')`, gated behind
   `isPlatformBrowser(inject(PLATFORM_ID))`, so the module is never
   `require()`d in Node at all.
2. **Unguarded browser APIs in root singletons.** `ThemeService` read/wrote
   `localStorage` directly. `SessionBootstrapService`'s blocking
   `APP_INITIALIZER` wired up a Firebase Auth listener, and injecting
   `AuthService` at all throws in Node unless `initializeApp()` has already
   run in that process. Both were fixed with `isPlatformBrowser` guards —
   `main.server.ts` now calls `initFirebaseApp()` before bootstrap
   (mirroring `main.ts`), and `SessionBootstrapService.start()` short-
   circuits to a resolved promise on the server, since the public pages
   never read `OrgContextService` anyway.

Any future root-level service or `APP_INITIALIZER` that touches
`window`/`document`/`localStorage`/Firebase at construction or init time
needs the same `isPlatformBrowser` treatment, or it will break the
prerender build (and therefore CI) even if it's only ever used from an
authenticated, CSR-only page.
