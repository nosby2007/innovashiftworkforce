# Internationalization (i18n)

Runtime translation via [Transloco](https://jsverse.github.io/transloco/) — a
user picks a language and the UI updates immediately, no rebuild or
separate URL per locale. This was chosen over Angular's built-in
`@angular/localize` specifically because this is a SaaS app with a
per-user language preference, not a marketing site that needs separate
indexed URLs per language.

## What's translated today

**Phase 1**
- **App shell chrome** — sidebar navigation, top bar, user menu, page
  titles. This is shared across every authenticated page, so it's the
  highest-leverage thing to translate first: even before an individual
  page's content is translated, all the navigation around it already is.
- **Employee Dashboard** (`/app/dashboard`) — welcome banner, the premium
  quick-link strip, and the three main cards (Manage My Schedule, My
  Notifications, My Schedule).

**Phase 2**
- **My Schedule** (`/app/schedule`) — header, request menu, mini calendar
  (including weekday abbreviations, which are a translated JSON array
  rather than individual keys — see `schedule.weekdaysShort`), filters,
  shift list, detail panel, and the "Share Shift to Marketplace" dialog.
- **Attendance** (`/app/attendance`) — page header, Current Shift panel,
  My Upcoming Shifts + Call Out form, Manual Punch panel, the 4 KPI
  cards, Recent Punches table, and the My Correction Requests panel
  including the inline correction-request form. **Not translated**: the
  dense ADP-style "Timecard Inquiry" ledger table at the top of the page
  and the read-only Corrections history table — both are secondary,
  lower-traffic detailed views, left for a follow-up pass.
- **My Payroll** (`/app/payroll`) — header, KPI cards, Pay Statement and
  Pay Code Breakdown cards, and the payroll rows table headers. **Not
  translated**: a few TypeScript-generated fallback strings inside table
  rows (e.g. "Assigned shift", "Open" for an unclosed punch, "approved
  leave" status text) — translating those requires injecting
  `TranslocoService` into row-construction logic and isn't wired to
  re-render on a language switch the way the `transloco` pipe is, so it
  was left out rather than shipping a half-reactive translation.

**Phase 3**
- **Admin Employees** (`/admin/employees`) — page header, search/role
  filters, no-org and empty states, employee cards (role labels, badges,
  Details/Request Transfer/Suspend/Revoke actions, Active/Inactive status),
  the Invite Employee drawer, and the Request Transfer drawer. **Not
  translated**: the toast/confirm-dialog strings in `submitInvite()`,
  `copyInviteLink()`, `submitTransferRequest()`, and `manageMembership()`
  (e.g. `window.confirm(...)` text and success/error toasts) — same
  TS-generated-string boundary as My Payroll's row fallbacks.
- **Admin Timesheets** (`/admin/timesheets`) — filters, the Attendance
  Management correction-fix panel, table headers, and empty state. **Not
  translated**: `breakLabel` ("On break" / "N min"), the raw
  `exceptionStatus` value shown via `titlecase`, and the toast/confirm/
  prompt strings in `applyFix()`, `rejectFix()`, and `deleteEntry()`.
- **Admin PTO** (`/admin/pto`) — hero, stats, filters, request queue,
  decision form, and the Payroll Impact panel. **Not translated**: the raw
  `r.requestType` / `r.status` values shown via `uppercase`/`titlecase`,
  and the success/error toasts in `approve()`/`reject()`.
- **Admin Payroll** (`/admin/payroll`) — hero, KPIs, Draft Payroll Run
  card (including the finalize/reopen flow), Payroll Readiness card,
  Employee Payroll Summary table, and Payroll Detail Rows table headers.
  **Not translated**: the Payroll Detail Rows table's row-level data
  (`shiftTitle` suffixes like " (Overtime)"/" (Holiday)", `checkOut`'s
  "Open" fallback, leave rows' "`{TYPE}` approved leave" / "approved
  leave" / "holiday pay" status text, and `finalizedAtLabel()`'s
  "recently" fallback) — all TS-generated inside row-construction logic,
  the same boundary as My Payroll.

**Phase 4**
- **Staff Onboarding** (`/app/onboarding`) — hero, summary cards, and the
  checklist steps (step titles/descriptions/actions are stored as
  translation keys in the `steps()` array and resolved with `| transloco`
  at the template site, same pattern as `NavItem.label`).
- **My Availability** (`/app/availability`) — form, upcoming list, and
  toast messages (translated via `TranslocoService.translate()` injected
  into the component — small page, low effort to cover fully).
- **My Accruals** (`/app/accruals`) — balances, time-off request form,
  request/ledger history, and toasts. `requestLabel()` returns a
  translation key resolved at the template site; `ledgerLabel()`'s
  dynamic `${TYPE} accrual` string is **not translated** (built from
  arbitrary ledger `type` data).
- **Message Center** (`/app/messages`) — chrome and toasts. Message
  `title`/`body` are **not translated** — that's org-authored content,
  not UI copy.
- **Shift Live Chat** (`/app/shift-chat`) — chrome and toasts. Chat
  `message` text and `senderRole` are **not translated** — user-generated
  content and raw role data, not UI copy.
- **Notifications** (`/app/notifications`) — control center chrome,
  category list (labels stored as keys, same array pattern as above),
  filters, detail panel, and toasts. Notification `title`/`body` are
  **not translated** — server-generated content, not UI copy.
- **Document Center** (`/app/documents`) — hero, profile readiness,
  checklist, app tiles, upload form, document history, and employment
  record. `completion().missing` (dynamic field names from
  `profile-completion.util.ts`) and `docLabel()` (from
  `EmployeeDocumentsRepo.labelFor()`) are **not translated** — both are
  generated outside this component.
- **Settings** (`/app/settings` and `/admin/settings`, shared component)
  — Appearance and Security panels. `ThemeService`'s theme
  `label`/`description` fields were changed to translation keys
  (`theme.*` namespace) since they're only consumed here.
- **My Profile** (`/app/profile`) — the entire page: header, launchpad,
  Personal Information, Team, Address, Tax Forms (W-4/W-2), Direct
  Deposit, Dependents, User Preferences, biometric Security toggle, and
  Two-Factor Authentication, plus every save/toast message. This is the
  largest fully-covered page — nothing was deferred except the raw
  `timezones` list (IANA zone names aren't translated, they're identifiers).
- **Shift Marketplace** (`/app/marketplace`) — header, tip card, current-
  shift banner, summary KPIs, search/filters, and all four board views
  (Available, Requests, Activity, Approvals), including
  `matchBadge()`/`claimLabel()`/`swapKindLabel()`/`swapCounterparty()`
  returning translation keys resolved at the template site. **Not
  translated**: the raw `r.status` swap-request badge text and
  `fmtActivity()`'s compact relative-time suffixes (`now`/`5m`/`2h`/`3d`)
  — both are short data-driven/format strings outside the page's static
  copy.

**Phase 5**
- **Admin Scheduler** (`/admin/scheduler`) — hero, planning KPIs, calendar
  toolbar/filters/legend, the Shift Actions and Staff Picker modals, and
  the Create/Edit Shift drawer, plus every toast message. This completes
  translation coverage of every authenticated in-app page. **Not
  translated**: the FullCalendar `eventContent()`/`eventClassNames()`
  callbacks, which build raw HTML strings outside Angular's template
  system — FullCalendar calls them directly on render, so they can't use
  the `transloco` pipe and wouldn't react to a language switch on
  already-rendered events; and the raw `statusOptions` filter `<option>`
  values (lowercase enum values shown as-is, not passed through the
  humanized `statusLegend` labels).
- **Public marketing site** (`/`, `/features`, `/pricing`, `/contact`,
  and the shared `PublicLayoutComponent` nav/footer) — every page fully
  translated, with the `LanguageSwitcherComponent` added to the public
  nav bar so visitors can switch before signing in. **Known limitation**:
  the site is client-side rendered with no prerendering (see the SEO
  foundation work), so there's no server-rendered French HTML for search
  engines to index and no separate `/fr/...` URLs — the switcher changes
  what a visitor sees in their browser, but does not give French content
  its own indexable, crawlable URL. Real multi-locale SEO (hreflang,
  per-locale prerendered routes) would require Angular SSR/prerendering,
  which is out of scope here.

**Everything is now translated except the public marketing site's SEO
metadata** (`<title>`/meta description set via `SeoService.setPage()`),
which stays English-only — it's read once per page load rather than
re-rendered on a language switch, and real SEO value from translating it
requires the prerendering work noted above.

## Languages

Currently `en` (default) and `fr`. Files live at
`frontend-angular/src/assets/i18n/{lang}.json`.

## How language is chosen

`LanguageService` (`core/i18n/language.service.ts`) resolves the active
language in this order:
1. `localStorage` (`isw_lang`) — works instantly, before auth resolves.
2. The signed-in user's saved `preferences.language` on their
   `orgs/{orgId}/users/{uid}` doc — authoritative, syncs across devices,
   loaded once per session after auth resolves.
3. Browser language (`navigator.language`).
4. `en`.

Switching language (via `LanguageSwitcherComponent`, in the app shell top
bar) updates both localStorage and the user's profile doc immediately.

## Adding a translation key

1. Add the English string to `en.json` under an appropriate namespace
   (`common`, `shell`, `nav`, `routeTitles`, `dashboard`, or a new one
   named after the page/feature).
2. Add the French translation to `fr.json` under the same key.
3. In the template: `{{ 'namespace.key' | transloco }}`. For
   interpolated values: `{{ 'namespace.key' | transloco: { name: value } }}`
   with `{{name}}` in the JSON string. Import `TranslocoModule` into the
   component's `imports` array.
4. For a value used in a property binding (e.g. `matTooltip`,
   `aria-label`) rather than interpolation:
   `[matTooltip]="'namespace.key' | transloco"`.
5. For a plain-object array field (like `NavItem.label` in the app
   shell) that gets interpolated at a different template site: store the
   translation *key* in the object, resolve it with `| transloco` at the
   usage site — don't bake the resolved string into the data structure,
   or it won't react to a language change.

## Adding a new language

1. Copy `en.json` to `{code}.json` in `assets/i18n/` and translate every
   value.
2. Add the code to `availableLangs` in `provideTransloco()`
   (`app.config.ts`) and to `LanguageService.SUPPORTED` and
   `LanguageService.options` (`core/i18n/language.service.ts`).
3. If the language is right-to-left (e.g. Arabic), that's a bigger,
   separate change — Transloco itself doesn't handle RTL layout; it
   needs `dir="rtl"` toggling and CSS logical-property review across the
   app, not just new translation strings.
