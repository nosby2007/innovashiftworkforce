# Internationalization (i18n)

Runtime translation via [Transloco](https://jsverse.github.io/transloco/) — a
user picks a language and the UI updates immediately, no rebuild or
separate URL per locale. This was chosen over Angular's built-in
`@angular/localize` specifically because this is a SaaS app with a
per-user language preference, not a marketing site that needs separate
indexed URLs per language.

## What's translated today (Phase 1)

- **App shell chrome** — sidebar navigation, top bar, user menu, page
  titles. This is shared across every authenticated page, so it's the
  highest-leverage thing to translate first: even before an individual
  page's content is translated, all the navigation around it already is.
- **Employee Dashboard** (`/app/dashboard`) — fully translated as the
  proof-of-concept for a full page: welcome banner, the premium quick-link
  strip, and the three main cards (Manage My Schedule, My Notifications,
  My Schedule).

**Everything else — Scheduler, Timesheets, Payroll, Admin pages, PTO,
Documents, Messages, the public marketing site, etc. — is still
English-only.** This was a deliberate phased start (infrastructure +
shell + one full pilot page), not an oversight. Translating the rest is
straightforward mechanical work following the pattern below, page by page.

## Languages

Currently `en` (default) and `fr`. Files live at
`frontend-angular/src/assets/i18n/{lang}.json`.

## How language is chosen

`LanguageService` (`core/i18n/language.service.ts`) resolves the active
language in this order:
1. `localStorage` (`isw_lang`) — works instantly, before auth resolves.
2. The signed-in user's saved `preferences.language` on their
   `orgs/{orgId}/users/{uid}` doc — authoritative, syncs across devices,
   loaded once per session after auth resolves.
3. Browser language (`navigator.language`).
4. `en`.

Switching language (via `LanguageSwitcherComponent`, in the app shell top
bar) updates both localStorage and the user's profile doc immediately.

## Adding a translation key

1. Add the English string to `en.json` under an appropriate namespace
   (`common`, `shell`, `nav`, `routeTitles`, `dashboard`, or a new one
   named after the page/feature).
2. Add the French translation to `fr.json` under the same key.
3. In the template: `{{ 'namespace.key' | transloco }}`. For
   interpolated values: `{{ 'namespace.key' | transloco: { name: value } }}`
   with `{{name}}` in the JSON string. Import `TranslocoModule` into the
   component's `imports` array.
4. For a value used in a property binding (e.g. `matTooltip`,
   `aria-label`) rather than interpolation:
   `[matTooltip]="'namespace.key' | transloco"`.
5. For a plain-object array field (like `NavItem.label` in the app
   shell) that gets interpolated at a different template site: store the
   translation *key* in the object, resolve it with `| transloco` at the
   usage site — don't bake the resolved string into the data structure,
   or it won't react to a language change.

## Adding a new language

1. Copy `en.json` to `{code}.json` in `assets/i18n/` and translate every
   value.
2. Add the code to `availableLangs` in `provideTransloco()`
   (`app.config.ts`) and to `LanguageService.SUPPORTED` and
   `LanguageService.options` (`core/i18n/language.service.ts`).
3. If the language is right-to-left (e.g. Arabic), that's a bigger,
   separate change — Transloco itself doesn't handle RTL layout; it
   needs `dir="rtl"` toggling and CSS logical-property review across the
   app, not just new translation strings.
